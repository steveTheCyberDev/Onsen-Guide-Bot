# Strategic Commit Workflow

When this skill is invoked, follow these steps strictly to ensure clean version control:

1. **Analyze Working Tree**: Run `git status` and check all unstaged/staged file modifications. If on the default branch (`main`), create a feature branch first.
2. **Group by Strategy**: Group the files into separate logical changes rather than making one massive "mega-commit". For example:
   - Group 1: Core backend logic changes (e.g., routes, services, agent)
   - Group 2: User interface or style updates (e.g., components, CSS)
   - Group 3: Tests or documentation updates (e.g., test suites, README, CLAUDE.md)
   - Group 4: Config / tooling (e.g., requirements.txt, package.json, pytest.ini)
3. **Verify and Test**: For each group, run the relevant linter and test suite before staging to ensure no broken code is committed.
   - Backend: `cd backend && .venv/bin/python -m pytest tests/ -q`
   - Frontend: `cd frontend && npm run build` (and lint if configured)
4. **Stage Incrementally**: Use `git add` to stage *only* the files belonging to the current logical group.
5. **Generate Conventional Message**: Write a concise message following the Conventional Commits specification:
   - Structure: `<type>(<optional scope>): <description>`
   - Types: feat, fix, refactor, docs, style, test, chore
   - Use the lowercase imperative mood (e.g., "feat(auth): add JWT token rotation").
   - Do NOT include marketing/"Generated by AI" footers.
   - DO end the message with the required trailer: `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`
6. **Commit and Repeat**: Run `git commit`. Then repeat the staging and committing steps for the remaining groups until the working tree is clean.
7. **Confirm Before Push**: Only push when the user explicitly asks. Report the resulting commits (hashes + messages) so the user can review the grouping.
